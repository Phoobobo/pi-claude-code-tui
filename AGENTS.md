# Repository Guidelines

## Project overview

`pi-claude-code-tui` is a pi package that customizes the pi TUI startup look:

- Extension entrypoint: `extensions/claude-code-startup.ts`
- Pure layout/editor helpers: `extensions/render-utils.ts`
- Unit tests: `tests/render-utils.test.ts`
- Screenshot asset: `assets/screenshot.png`
- Package metadata and pi extension registration: `package.json`
- CI: `.github/workflows/ci.yml` (typecheck + tests)
- npm publishing workflow: `.github/workflows/publish.yml`

The extension registers on `session_start`, replaces the TUI header/editor, and provides:

- `/use-claude-code-tui` to switch to this package's custom TUI
- `/use-default-tui` to switch back to pi's built-in TUI

## Development commands

- Type check: `npx tsc --noEmit` (or `npm run typecheck`)
- Tests: `npm test`
- Run locally in pi: `pi -e .`
- Preview npm package contents: `npm pack --dry-run`

Run `npx tsc --noEmit` and `npm test` before committing TypeScript changes.

Pure layout/editor helpers live in `extensions/render-utils.ts` (not a pi entrypoint).
The registered extension is only `extensions/claude-code-startup.ts` via `package.json` `pi.extensions`.

## Coding style

- TypeScript ESM (`"type": "module"`, `moduleResolution: "NodeNext"`).
- Keep `strict` compatibility.
- Existing source uses tabs for indentation.
- Prefer small helper functions for terminal width, padding, truncation, and ANSI styling.
- Preserve pi defaults by passing `undefined` for built-in footer/spinner/editor/header when restoring.

## Assets

- The package image is `assets/screenshot.png`.
- `package.json` points `pi.image` at the raw GitHub URL for this asset on `main`.
- When replacing the screenshot, verify with `file assets/screenshot.png` and include it in the commit.

## Release and publishing

Publishing is handled by GitHub Actions in `.github/workflows/publish.yml`.

Typical release flow:

1. Update `package.json` version.
2. Run `npx tsc --noEmit`, `npm test`, and `npm pack --dry-run`.
3. Commit and push to `main`.
4. Create and push tag `vX.Y.Z`.
5. Create GitHub release `vX.Y.Z`.
6. The release workflow publishes to npm with provenance using `NPM_TOKEN`.

If npm has already been published manually or via `workflow_dispatch`, avoid triggering a duplicate release publish. One safe approach is to temporarily disable the publish workflow, create the GitHub release, then re-enable the workflow.

## Notes

- `node_modules/`, `dist/`, logs, and `.DS_Store` are ignored.
- A local `package-lock.json` may be generated by `npm install`; do not commit it unless the project intentionally starts tracking lockfiles.
